# 레저렉션(Resurrection) PRD

## 1. 문서 개요
- 문서명: Product Requirements Document (PRD) - 레저렉션
- 문서 버전: v1.3
- 최종 수정일: 2026-03-14
- 기준 저장소: `SProj.Resurrection`
- 현재 UI 표기 앱 버전: `v0.3.0`
- 대상 플랫폼: Windows x64

## 2. 제품 한 줄 정의
레저렉션은 멀티 모니터 환경에서 무너진 창 배치를 저장하고 빠르게 복원하는 Windows 데스크톱 앱이다.

## 3. 배경과 문제
- 도킹/언도킹, 모니터 전원 변경, 해상도 변경 이후 창 위치와 크기가 자주 깨진다.
- 개발자/디자이너/사무 사용자는 같은 창 배치를 반복적으로 다시 맞추는 비용이 크다.
- 수동 정렬 비용을 줄이기 위해 창 배치를 저장하고, 모니터 변화 시 자동 복원하는 기능이 필요하다.

## 4. 제품 목표
- 사용자가 창 재배치에 쓰는 시간을 줄인다.
- 수동 복원뿐 아니라 자동 복원을 통해 반복 작업을 최소화한다.
- 초보 사용자도 1분 내 핵심 기능(저장/복원/삭제/설정)을 사용할 수 있게 한다.

## 5. 적용 범위
### 5.1 포함 범위 (In Scope)
- 현재 열린 창 스냅샷 저장
- 저장된 레이아웃 목록 조회/수동 복원/삭제
- 현재 열린 창 목록 미리보기
- 일반 창/시스템 창 분리 미리보기
- 모니터 변경 이벤트 감지 후 자동 복원
- 모니터 컨텍스트(Strict/Fuzzy) 기반 레이아웃 선택
- 복원 전 트레이 확인 풍선(옵션)
- 테마 설정(다크/라이트/시스템)
- 트레이 상주 실행
- 로컬 저장소 기반 설정/레이아웃 영속화

### 5.2 제외 범위 (Out of Scope)
- 미실행 앱을 자동 실행해 복원하는 기능
- 계정 동기화 및 클라우드 백업
- 팀 단위 레이아웃 공유
- macOS/Linux 배포

## 6. 최신 변경사항 반영 (이번 업데이트 기준)
### 6.1 기능 추가 및 고도화
- `monitorContext`를 레이아웃 저장 시 함께 기록한다.
- `lastRestoredByMonitorKey` 맵을 도입해 모니터 조합별 마지막 복원 레이아웃을 관리한다.
- 자동 복원 대상 조회 순서를 `strictKey -> fuzzyKey -> (레거시 fallback) lastLayoutId`로 구성했다.
- 복원 모드를 `manual`/`auto`로 분리하고 모드별 재시도 정책을 다르게 적용했다.
- 복원 결과에 창 단위 상태와 실패 사유를 포함하는 리포트를 제공한다.
- 시스템 창(예: 검색, 입력 경험, 일부 Shell 창)을 분리/제외해 저장 및 복원 정확도를 높였다.

### 6.2 자동 복원 안정성 개선
- 디스플레이 변경 이벤트를 디바운스 처리한다.
- 동일 모니터 키에서 짧은 시간 내 중복 자동 복원을 억제한다.
- 복원 진행 중 새 이벤트가 들어오면 in-flight 가드로 충돌을 방지한다.
- latest-only 처리로 마지막 이벤트 기준 복원만 실행한다.

### 6.3 UI/UX 변경
- 메인 UI를 3개 탭으로 재구성했다.
- 탭 1: 레이아웃 저장/목록/복원/삭제
- 탭 2: 현재 창 미리보기(일반 창/시스템 창 분리)
- 탭 3: 설정(테마/자동 복원/복원 확인/앱 정보)
- 레이아웃 항목 호버 시 저장된 창 목록과 모니터 상세 정보를 팝업으로 보여준다.
- 복원 상태/오류를 하단 상태 메시지로 피드백한다.
- 트레이 상주 동작과 재열기 흐름을 UI 안내 문구로 명확히 했다.

### 6.4 라이트 모드 추가
- 테마 모드 `dark`, `light`, `system` 3가지를 지원한다.
- `system` 모드에서 OS 색상 선호(`prefers-color-scheme`) 변경을 실시간 반영한다.
- CSS 변수 기반 토큰(`data-theme`)으로 전체 컴포넌트 색상을 일관되게 전환한다.

### 6.5 데이터/호환성 변경
- 설정 초기값에 `themeMode`, `lastRestoredByMonitorKey`를 추가했다.
- 기존 저장 데이터와의 호환을 위해 설정 정규화 로직을 적용했다.
- 구버전 레이아웃에 시스템 창이 포함된 경우 읽기 시 자동 정리한다.

## 7. 상세 기능 요구사항
### FR-01 레이아웃 저장
- 사용자는 레이아웃 이름을 입력해 현재 창 배치를 저장할 수 있어야 한다.
- 저장 시 일반 창만 저장해야 하며 시스템 창은 저장 대상에서 제외한다.
- 저장 결과는 즉시 목록에 반영되어야 한다.

### FR-02 레이아웃 목록 조회
- 저장된 레이아웃은 생성 시각과 창 개수와 함께 보여야 한다.
- 마지막 복원 레이아웃은 목록에서 시각적으로 구분되어야 한다.
- 레이아웃마다 연결된 모니터 정보를 함께 표시해야 한다.

### FR-03 레이아웃 수동 복원
- 사용자는 목록에서 레이아웃을 선택해 복원할 수 있어야 한다.
- 복원 결과는 성공 여부와 복원된 창 개수로 피드백되어야 한다.
- 복원 실패 시 사용자에게 오류 메시지를 보여야 한다.

### FR-04 레이아웃 삭제
- 사용자는 저장된 레이아웃을 삭제할 수 있어야 한다.
- 삭제 전 확인 절차가 있어야 한다.
- 삭제된 레이아웃이 마지막 복원 대상으로 기록되어 있다면 참조를 정리해야 한다.

### FR-05 현재 창 미리보기
- 사용자는 현재 열린 창 목록을 수동 새로고침으로 조회할 수 있어야 한다.
- 일반 창과 시스템 창을 분리해 보여야 한다.
- 시스템 창 목록은 기본 접힘 상태로 제공해야 한다.

### FR-06 모니터 컨텍스트 수집
- 레이아웃 저장 시 모니터 컨텍스트를 함께 저장해야 한다.
- 컨텍스트에는 Strict/Fuzzy 키와 모니터 메타데이터가 포함되어야 한다.
- 모니터 모델명이 불명확한 경우 해상도/ID 기반 대체 라벨을 생성해야 한다.

### FR-07 자동 복원
- 모니터 추가/제거/메트릭 변경 이벤트에서 자동 복원을 시도해야 한다.
- 자동 복원은 사용자 설정(`autoRestore`)이 켜져 있을 때만 동작해야 한다.
- 자동 복원은 모니터 컨텍스트 키 우선순위에 따라 레이아웃을 선택해야 한다.

### FR-08 복원 확인 UX
- 사용자 설정(`askBeforeRestore`)이 켜져 있으면 복원 전 트레이 확인 풍선을 노출해야 한다.
- 풍선 클릭 시 복원을 진행하고, 닫힘/만료 시 복원을 취소해야 한다.

### FR-09 테마 설정
- 사용자는 `시스템 자동`, `다크`, `라이트` 중 하나를 선택할 수 있어야 한다.
- 선택된 테마는 저장되어 앱 재실행 후에도 유지되어야 한다.
- 시스템 자동 모드는 OS 테마 변경을 실시간 반영해야 한다.

### FR-10 트레이 상주
- 메인 창 닫기 동작은 앱 종료가 아니라 트레이 숨김이어야 한다.
- 트레이 메뉴에서 앱 다시 열기와 종료를 제공해야 한다.

### FR-11 복원 매칭/재시도
- 복원은 제목/프로세스/실행 파일/모니터 힌트 기반 점수 매칭으로 수행해야 한다.
- 애매한 매칭은 실패 처리해 잘못된 창 이동을 줄여야 한다.
- 복원은 재시도 및 이동 검증을 수행해야 한다.

### FR-12 보안 IPC 통신
- 렌더러는 preload를 통해서만 IPC를 호출해야 한다.
- 메인 프로세스는 정의된 IPC 핸들러만 노출해야 한다.

## 8. 데이터 모델 요구사항
### 8.1 Layout
- `id: string`
- `name: string`
- `windows: WindowInfo[]`
- `createdAt: number`
- `monitorContext?: MonitorContext`

### 8.2 WindowInfo
- `windowId?: number`
- `processId?: number`
- `exePath?: string`
- `processName: string`
- `rawTitle?: string`
- `normalizedTitle?: string`
- `title: string`
- `x, y, width, height: number`
- `capturedAt?: number`
- `displayHint?: { monitorId?: number; monitorModel?: string }`
- `isSystemWindow?: boolean`

### 8.3 MonitorContext
- `strictKey: string`
- `fuzzyKey: string`
- `monitorModels: string[]`
- `monitorSignatures: string[]`
- `capturedAt: number`
- `monitors?: monitor 상세 배열`

### 8.4 Settings
- `autoRestore: boolean`
- `askBeforeRestore: boolean`
- `themeMode: 'dark' | 'light' | 'system'`
- `lastRestoredByMonitorKey: Record<string, string>`
- `lastLayoutId: string | null` (레거시 fallback)

## 9. 비기능 요구사항
- 성능: 사용자 체감 기준 저장/목록 조회/수동 복원 요청은 즉시 반응해야 한다.
- 안정성: 네이티브 모듈 로딩 실패 시 앱 전체가 크래시하지 않아야 한다.
- 정확도: 시스템 창 제외 및 점수 매칭으로 오복원 확률을 낮춰야 한다.
- 보안: `contextIsolation: true`, `nodeIntegration: false`, preload IPC 브리지 구조를 유지해야 한다.
- 호환성: Windows x64 배포와 설치 흐름(NSIS)을 제공해야 한다.

## 11. 사용자 시나리오
1. 사용자가 작업용 창 배치를 만든 뒤 레이아웃을 저장한다.
2. 도킹/언도킹 등으로 모니터 구성이 바뀐다.
3. 자동 복원이 켜져 있으면 모니터 컨텍스트에 맞는 레이아웃을 자동으로 찾는다.
4. 복원 확인이 켜져 있으면 트레이 풍선 클릭 후 복원한다.
5. 사용자는 기존 작업 배치를 빠르게 회복한다.

## 12. 성공 지표 (KPI)
- 복원 성공률: 복원 시도 대비 1개 이상 창 복원 성공 비율
- 시간 절감 체감: 수동 창 정렬 시간 감소에 대한 사용자 응답
- 반복 사용률: 주간 레이아웃 복원 사용 빈도
- 안정성: 비정상 종료율 및 자동 복원 실패율

## 13. 리스크 및 대응
- 리스크: 제목이 유사한 창이 많을 때 잘못 매칭될 수 있다.
- 대응: 점수 임계치, 애매함(ambiguous) 실패 처리, 재시도 전략을 유지한다.
- 리스크: 모니터 이벤트가 연속 발생하면 중복 복원이 발생할 수 있다.
- 대응: 디바운스, 중복 억제, in-flight 가드를 유지한다.
- 리스크: 네이티브 모듈 의존성 문제로 기능 저하가 발생할 수 있다.
- 대응: 로드 실패 시 격리 처리와 로그 수집을 유지한다.

## 14. 검증 체크리스트
- 레이아웃 저장/복원/삭제 흐름이 정상 동작한다.
- 자동 복원 ON/OFF, 복원 확인 ON/OFF 조합이 의도대로 동작한다.
- 모니터 컨텍스트 키(Strict/Fuzzy) 기반 선택이 정상 동작한다.
- 시스템 창 분리가 정상 동작하고 저장 대상에서 제외된다.
- 다크/라이트/시스템 테마 전환이 즉시 반영된다.
- `npm run build` 및 패키징 결과물이 정상 생성된다.
